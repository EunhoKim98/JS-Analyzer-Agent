JS Analyzer Agent · v0.1 · 솔루션 설명

결정론적 정적분석이 후보를 좁히고,
LLM 에이전트가 그것만 추론한다

웹페이지·번들 자바스크립트에서 취약점을 찾는 멀티 에이전트 CLI 솔루션. 값싸고 반복 가능한 정적 분석으로 의심 지점(sink)을 먼저 추려낸 뒤, LLM은 순위가 매겨진 후보에 대해서만 OODA 추론과 오탐 판정을 수행합니다. 순수 LLM 스캔보다 정밀하고, 순수 정적 분석보다 문맥을 이해합니다.

CLI · TypeScript 엔진 Babel AST + Playwright 에이전트 Sonnet / Opus GitHub ↗
01 — 개요

무엇을 푸는 솔루션인가

현대 웹 프런트엔드는 번들링·난독화된 대용량 자바스크립트로 배포됩니다. 사람이 일일이 읽기 어렵고, LLM에 통째로 넣으면 비싸고 부정확합니다. JS Analyzer Agent는 “먼저 값싸게 좁히고, 비싼 추론은 후보에만” 이라는 원칙으로 이 문제를 나눕니다. 파일·디렉터리·URL을 입력하면 취약점 후보를 찾아 분석하고, 사람이 읽는 HTML 리포트와 기계가 읽는 JSON을 runs/<run_id>/에 남깁니다.

02 — 핵심 컨셉

왜 이렇게 나눴나

A 정적 프리패스

AST로 source·sink·sanitizer를 찾고 휴리스틱 테인트로 위험 지점을 순위화. LLM 호출 없이 어디서든 실행되는 결정론적 단계.

B 에이전트 추론

순위가 매겨진 sink마다 Sonnet이 한 번의 OODA 라운드로 실제 도달 가능성·악용 경로를 분석. 코드 전체가 아닌 후보에만 집중.

C 오탐 판정

Opus가 각 finding을 재검토해 false positive를 걸러내고, 결정론적 recheck로 판정을 교차 검증. 신뢰도 높은 결과만 남깁니다.

D 데이터 기반 규칙

취약점 클래스는 rules/*.yaml 카드로 정의. 클래스 추가 = 카드 추가. 코드 수정 없이 커버리지 확장.

03 — 파이프라인

수집에서 리포트까지

결정론적 (LLM 없음) LLM 에이전트 (API 키 필요) 이후 마일스톤
0
ingest deterministic

파일 · 디렉터리 · URL을 입력받아 content-hash로 정규화. 페이지 URL이면 다음 단계에서 스크립트를 수집합니다.

1
unbundle deterministic

소스맵이 있으면 원본으로 복원, 없으면 beautify. 헤드리스 크로미움(Playwright)으로 외부·인라인·동적 주입 스크립트까지 확보합니다.

2
static deterministic

Babel AST로 sink·source·sanitizer 탐지 + 휴리스틱 테인트 + 에셋·시크릿 수집. LLM 없이 sink에 순위를 매깁니다.

5
analyze LLM agent Sonnet

순위가 매겨진 sink마다 한 번의 OODA 라운드로 도달 가능성과 악용 경로를 추론합니다.

OODA Observe→Orient→Decide→Act ↻ per ranked sink
7
judge LLM agent Opus

false positive 판정 + 결정론적 recheck 교차 검증. 확신 있는 finding만 통과시킵니다.

8
report deterministic

report.html + assets_export.json 생성. 결과는 runs/<run_id>/에 sinks·findings·verdicts·trace와 함께 저장됩니다.

3·4·6

번호 공백은 로드맵. explore / asset / chaining 에이전트, 능동 PoC 검증, Burp 확장, Claude Code 래퍼가 이후 마일스톤으로 채워집니다 (DESIGN.md).

04 — 탐지 모델 & 산출물

테인트 모델과 출력

테인트 흐름 source → sink

source → sanitizer? → sink

사용자 입력(source)이 살균(sanitizer) 없이 위험 함수(sink)에 도달하면 취약점 후보로 봅니다. v0.1 시드 클래스:

DOM-XSSopen redirectpostMessageprototype pollutionhardcoded secret

산출물 runs/<run_id>/

  • sinks.json순위화된 의심 지점
  • asset_manifest.json수집된 에셋 목록
  • findings.jsonl에이전트 분석 결과
  • verdicts.json오탐 판정
  • report.html사람이 읽는 리포트
  • trace/에이전트별 로그
05 — 사용법

설치와 실행

# 설치 & 빌드
git clone https://github.com/EunhoKim98/JS-Analyzer-Agent.git
cd JS-Analyzer-Agent
npm install
npm run build
npx playwright install chromium   # 페이지 수집용(1회)
# 정적 분석만 — API 호출 없이 어디서든 실행
npx tsx src/cli.ts analyze samples/vulnerable.js --no-llm

# 전체 파이프라인 (정적 + LLM 분석 + 오탐 판정) — 인증 필요
npx tsx src/cli.ts analyze samples/vulnerable.js

# 페이지 URL → 헤드리스 크로미움으로 JS 수집 후 분석
node dist/cli.js analyze https://example.com --no-llm
node dist/cli.js analyze https://example.com --all-hosts   # 서드파티 JS 포함
06 — 인증

API 키 또는 구독 토큰

LLM 단계(analyze · judge)는 아래 순서로 자격증명을 찾습니다. 먼저 매칭되는 것이 사용됩니다.

1
ANTHROPIC_API_KEY

Anthropic API 키 (사용량 과금).

2
CLAUDE_CODE_OAUTH_TOKEN / ANTHROPIC_AUTH_TOKEN

Claude Pro/Max 구독 OAuth 토큰 (claude setup-token).

3
~/.claude/.credentials.json

claude login 후 자동 감지 (Windows/Linux). macOS 키체인 자격증명은 자동으로 읽지 않으니 토큰을 export 하세요.

07 — 로드맵

v0.1 이후

파이프라인 번호의 공백(3·4·6)은 다음 마일스톤을 위한 자리입니다: explore / asset / chaining 에이전트, 능동 PoC 검증(실제 요청으로 취약점 재현), Burp 확장(프록시 트래픽에서 in-scope JS 필터링), Claude Code 래퍼. 상세 설계는 저장소의 DESIGN.md에 정리되어 있습니다.
